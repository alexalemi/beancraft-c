# Devices

A counter machine can't do I/O. beancraft cheats by reserving certain *register
names* as **device registers**: to the language they're ordinary registers, but
the runtime intercepts them. There are two kinds:

- **data registers** — plain storage the runtime reads or writes (`con/byte`,
  `screen/x`, `time/year`, …). `inc`/`deb` work normally.
- **trigger registers** — touching them performs an action *instead of* the
  normal effect:
  - **inc-triggers** (`con/emit`, `screen/plot`, `sys/halt`, …): `inc R` fires the
    side effect and does **not** increment `R`. (`deb R` on one is a no-op.)
  - **deb-polls** (`con/read`, `kbd/event`, `mouse/event`): `deb R` first lets
    the device refresh its data registers, *then* decrements/branches as usual —
    so the device hands you a "got something?" flag via `R`'s own value (`1` =
    yes, consumed by the `deb`; `0` = no).

Device registers are **global**: they're never renamed by `use`/`func`, so
`con/byte` always means the console's byte. If a program mentions a trigger
register, the registers it implicitly depends on are created automatically
(mentioning `con/emit` also creates `con/byte`; `screen/plot` creates
`screen/{x,y,color}`; etc.).

A program that touches any device runs **without the step cap** (so interactive
programs run until they exit) — unless you pass `-s N` explicitly.

## `sys/` — process control

| register | kind | effect |
| --- | --- | --- |
| `sys/halt` | inc-trigger | `exit(0)` immediately |
| `sys/code` | data | the exit status for `sys/exit` |
| `sys/exit` | inc-trigger | `exit(sys/code & 0x7f)` |
| `sys/debug` | inc-trigger | dump every register's value to stderr (for poking at a running program) |

## `con/` — console (stdin/stdout/stderr)

| register | kind | effect |
| --- | --- | --- |
| `con/byte` | data | the byte to print, or the byte just read |
| `con/emit` | inc-trigger | write `con/byte & 0xff` to **stdout**, then `con/byte := 0` |
| `con/err` | inc-trigger | same, to **stderr** |
| `con/read` | deb-poll | read one byte from **stdin** into `con/byte` and set `con/read := 1`; on EOF set `con/read := 0` (and leave `con/byte`). So `deb con/read got eof` is "read a byte; on success consume the flag and go to `got`, on EOF go to `eof`". |

To print a character: put its code in `con/byte` (e.g. with the standard
library's `addn N con/byte`), then `inc con/emit`. See `examples/hello.bc`,
`examples/cat.bc`, `examples/text.bc`.

## `time/` — wall clock

| register | kind | effect |
| --- | --- | --- |
| `time/now` | inc-trigger | sample the local time into the eight data registers below |
| `time/year` | data | full year (e.g. 2026) |
| `time/month` | data | 1–12 |
| `time/day` | data | 1–31 |
| `time/hour` | data | 0–23 |
| `time/min` | data | 0–59 |
| `time/sec` | data | 0–60 |
| `time/dow` | data | day of week, 0 = Sunday |
| `time/yday` | data | day of year, 1-based |

See `examples/clock.bc`, `examples/dayOfWeek.bc`.

## `rand/` — pseudo-random bytes

A 64-bit xorshift PRNG, seeded at startup from the monotonic clock + PID (so
runs differ); call `rand/setseed` for reproducibility.

| register | kind | effect |
| --- | --- | --- |
| `rand/next` | inc-trigger | `rand/byte := (next PRNG output) & 0xff` |
| `rand/byte` | data | the freshly generated byte, 0–255 |
| `rand/seed` | data | the seed value for `rand/setseed` |
| `rand/setseed` | inc-trigger | reseed the PRNG from `rand/seed` (a seed of 0 becomes 1) |

See `examples/rand.bc`, `examples/dvd.bc`.

## The framebuffer: `screen/` and `palette/`

If a program mentions any `screen/*` or `palette/*` register, the runtime brings
up a renderer. There's a single off-screen back buffer of palette indices
(8-bit); nothing appears until you `inc screen/flush`, which presents the buffer
and vsyncs to roughly 60 Hz. Two backends:

- **SDL** (build with `make sdl`, or `bccompile --sdl`): a 256×192 window,
  nearest-neighbour-scaled 3×, resizable. This is also where keyboard and mouse
  events come from. `screen/width`/`height` read 256 / 192.
- **terminal** (default build): the alternate screen buffer; each text cell shows
  *two stacked pixels* via the "▀" upper-half-block (top pixel = foreground,
  bottom = background). Size = your terminal's columns × (rows × 2), or 64 × 48
  if stdout isn't a tty. The original screen is restored on exit. Keyboard via
  raw stdin; no mouse.

### Coordinates & colour

| register | meaning |
| --- | --- |
| `screen/x`, `screen/y` | pixel coordinates (for plot/clear they wrap mod width/height) |
| `screen/color` | palette index, taken mod 256 |
| `screen/width`, `screen/height` | the back-buffer size; set by the runtime — read these, don't write them |

The palette defaults to the standard xterm 256-colour layout (16 system colours,
a 6×6×6 RGB cube, 24 greys). Override entries:

| register | meaning |
| --- | --- |
| `palette/index` | which palette slot (0–255) to set |
| `palette/r`, `palette/g`, `palette/b` | the RGB components (0–255) |
| `palette/set` (inc-trigger) | `palette[index] := rgb(r, g, b)` |

### Drawing primitives (all inc-triggers)

| register | effect |
| --- | --- |
| `screen/plot` | `framebuffer[y][x] := color` |
| `screen/clear` | fill the whole back buffer with palette index 0 |
| `screen/flush` | present the back buffer; vsync ~60 Hz; pump input events |
| `screen/sprite` | blit an 8×8 1-bit bitmap at (x, y): rows come from `screen/row0..row7` (one byte each, **bit 0 = leftmost pixel**); set bits draw `color`, clear bits are transparent; clipped at the edge (no wrap) |
| `screen/rect` | fill a `screen/rectw` × `screen/recth` rectangle at (x, y) with `color` (clipped to the screen, so huge sizes cost only the visible part) |
| `screen/text` | blit the built-in 8×8 font glyph for byte `screen/glyph` (printable ASCII 32–126; others draw blank) at (x, y) in `color`, then advance `screen/x += 8` and clear `screen/glyph` — so repeated `text` writes a string |
| `screen/sample` | the reverse of `plot`: `screen/pixel := framebuffer[y][x]` (read back a pixel — used for collision tests) |
| `screen/glyph`, `screen/pixel`, `screen/row0..row7`, `screen/rectw`, `screen/recth` | the data registers the above read |

See `examples/life.bc`, `examples/langton.bc`, `examples/sierpinski.bc`,
`examples/stars.bc`, `examples/gravity.bc`, `examples/dvd.bc`, `examples/bounce.bc`,
`examples/paint.bc`, `examples/pong.bc`, `examples/text.bc`.

## `kbd/` — keyboard

| register | kind | effect |
| --- | --- | --- |
| `kbd/event` | deb-poll | if a key is queued, pop it: `kbd/char := ASCII byte`, `kbd/code := key code`, and set `kbd/event := 1`; otherwise set `kbd/event := 0`. So `deb kbd/event got none` is "is there a key?". |
| `kbd/char` | data | the character (ASCII) of the popped key |
| `kbd/code` | data | a key code (same as `kbd/char` in the terminal backend; an SDL keycode in the SDL backend) |

In the terminal backend, keys are read from raw, non-blocking stdin (Ctrl-C
still quits). In the SDL backend they come from the window's key events.

## `mouse/` — pointer (SDL backend only)

| register | kind | effect |
| --- | --- | --- |
| `mouse/event` | deb-poll | if the pointer has moved or a button changed since the last poll, set `mouse/x`, `mouse/y`, `mouse/buttons` and `mouse/event := 1`; otherwise `mouse/event := 0` |
| `mouse/x`, `mouse/y` | data | pointer position in screen pixels |
| `mouse/buttons` | data | bitmask of held buttons |

Without the SDL build the mouse never reports anything (`deb mouse/event`
always takes the zero branch). See `examples/paint.bc`.

## `audio/` — square-wave beeper (SDL backend only)

| register | kind | effect |
| --- | --- | --- |
| `audio/freq` | data | tone frequency in Hz |
| `audio/duration` | data | tone length in milliseconds |
| `audio/play` | inc-trigger | queue a square-wave tone of `audio/freq` Hz for `audio/duration` ms |

Without the SDL build, `inc audio/play` is a harmless no-op. The standard
library's `note FREQ MS` wraps the three-register dance. See `examples/chime.bc`,
and the beeps in `examples/pong.bc`.

## Quick reference: which registers trigger something

- **inc-triggers** (fire on `inc`): `sys/halt`, `sys/exit`, `sys/debug`,
  `con/emit`, `con/err`, `time/now`, `rand/next`, `rand/setseed`, `palette/set`,
  `screen/plot`, `screen/clear`, `screen/flush`, `screen/sprite`, `screen/rect`,
  `screen/text`, `screen/sample`, `audio/play`.
- **deb-polls** (refresh-then-branch on `deb`): `con/read`, `kbd/event`,
  `mouse/event`.
- everything else listed above is a plain data register.

Note for the optimizer: the loop-folding passes (`-O`) never fold a loop over a
device register — `inc`/`deb` on one has observable side effects, so the loop
stays as written.
